## Why this matters

`dbos = ">=2.23.0"` has no upper bound, so the next `poetry update`
(the `mise run update` chore runs one) is free to pull in a breaking
3.x of a workflow engine that owns durable timers and its own system
database ŌĆö the kind of dependency where a silent major bump hurts
most. Next to it, `django-vite = "==3.1.0"` is the only exact pin in a
file of caret/tilde ranges, which blocks even patch fixes; nobody
remembers whether that freeze was deliberate. This plan bounds dbos to
`<3` and either documents or relaxes the django-vite pin, so future
dependency chores stay boring.

## Steps

### Step 1: Bound dbos

In `pyproject.toml:46`, change:

```toml
dbos = ">=2.23.0"
```

to:

```toml
dbos = ">=2.23.0,<3"
```

Then run `poetry lock` (plain ŌĆö Poetry 2 preserves locked versions by
default) and `poetry install`.

**Verify**: `grep -n 'dbos' pyproject.toml` ŌåÆ shows `">=2.23.0,<3"`;
`grep -A 1 '^name = "dbos"' poetry.lock` ŌåÆ `version = "2.26.0"`
(unchanged). If the locked version is anything other than `2.26.0`,
see STOP conditions.

### Step 2: Investigate the django-vite pin

Run:

```sh
git log --oneline -S "django-vite" -- pyproject.toml
git log --oneline -G "django-vite" -- pyproject.toml
grep -n -B 2 "django-vite" pyproject.toml
```

At planning time the history was grafted, so the `git log` commands
only return the graft commit `844db78` and prove nothing either way;
the grep confirms there is no explanatory comment. If your clone is
deeper and any commit message, comment, or linked PR shows the exact
pin was deliberate (a known 3.1.x regression, a compatibility freeze),
STOP and report ŌĆö keep the pin.

**Verify**: you have an explicit yes/no answer to "is the freeze
documented as intentional?" recorded for your report.

### Step 3: Relax django-vite (only if Step 2 found no rationale)

In `pyproject.toml:33`, change:

```toml
django-vite = "==3.1.0"
```

to:

```toml
django-vite = "~3.1.0"
```

(`~3.1.0` = `>=3.1.0,<3.2.0` ŌĆö patch updates only, matching the tilde
style of `authlib = "~1.7.0"` and `django = "~6.0.6"` in the same
block.) Run `poetry lock` and `poetry install` again.

**Verify**: `grep -A 1 '^name = "django-vite"' poetry.lock` ŌåÆ
`version = "3.1.0"` (a newer 3.1.x may appear only if 3.1.0 stopped
satisfying the constraint, which it does not ŌĆö so it must stay
3.1.0); `poetry check` ŌåÆ "All set!".
