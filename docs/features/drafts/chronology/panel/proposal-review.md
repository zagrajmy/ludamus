# Proposal review

## Triaging proposals

As a reviewer, I want to filter the proposals list by decision status, so
that I can focus on what still needs a decision.

As a reviewer, I want the proposals list to default to the queue of
undecided proposals, so that my work surface is in front of me when I
arrive.

As a reviewer, I want status filtering to combine with the other
proposal filters, so that I can narrow the queue along several axes at
once.

As a reviewer, I want to open a proposal in a view that shows every
field, so that I can read the whole submission before deciding.

As a reviewer, I want to accept or reject a proposal from where I am
reading it, so that I can decide without bouncing back to the list.

As a reviewer, I want to revisit a previously-decided proposal and
change my mind, so that I can correct mistakes.

As a reviewer, I want to land on the next proposal after I decide, so
that I can triage a queue without returning to the list each time.

As a reviewer, I want the "next" advance to respect the filters I am
using, so that triaging stays inside the queue I chose.

As a reviewer, I want to land back on the list when the queue is
exhausted, so that I know there is nothing left to triage.

## Downstream consequences

As an organiser, I want only accepted proposals offered for scheduling,
so that I cannot place items on the agenda that have not been approved.
